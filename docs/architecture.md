# Architecture

TRACE is a single Python process (`python3 -m f1trace`) with two threads
and a browser frontend:

```
                UDP 20777                      HTTP 8020
  F1 25 game ─────────────▶ Recorder thread ─┐   ┌─▶ Browser (static/)
  (or tools/fake_game.py)   packets.py        │   │
                            recorder.py       ▼   │
                                          SQLite (data/f1trace.db)
                                              ▲   │
                            Server thread ────┘───┘
                            server.py  (JSON API + static files)
```

- **Recorder thread** (`recorder.py`) — owns the UDP socket and the only
  *writer* DB connection. Parses packets, segments laps, writes one row
  per completed lap.
- **HTTP server** (`server.py`) — `ThreadingHTTPServer`; each request
  thread opens its own *read-only* SQLite connection. WAL mode makes the
  single-writer / many-readers split safe.
- **Viewer** (`static/`) — a single-page app, no build step, no
  framework; three `<canvas>` layers plus SVG for the HUD.

Everything is Python 3 standard library — no dependencies to install on
the machine that sits next to the console. The same constraint holds for
the frontend: plain JS, one file.

## Module map

| Path | Role |
| --- | --- |
| `f1trace/__main__.py` | CLI entry: prints game-settings banner, starts both threads |
| `f1trace/packets.py` | UDP struct layouts + pure parsers (2025 and 2026 formats) |
| `f1trace/recorder.py` | listener, per-car caches, lap segmentation, ghost handling, DB writes |
| `f1trace/db.py` | schema, migrations, sample blob pack/unpack |
| `f1trace/server.py` | JSON API + static file serving |
| `f1trace/ids.py` | enum id → name tables (tracks, session types, teams) |
| `f1trace/static/app.js` | the entire viewer |
| `f1trace/static/tracks.json` | real circuit outlines, generated by `tools/build_tracks.py` |
| `tools/fake_game.py` | synthetic game for end-to-end tests (reproduces ghost quirks) |
| `tools/build_tracks.py` | one-off: f1-circuits GeoJSON → `tracks.json` |

## Wire formats

`packets.py` supports two layouts, switched per packet by the
`packetFormat` header field: **2025** (base F1 25, 22 car slots) and
**2026** ("2026 Season Pack", 24 slots; g-forces become int16, engine
temp uint8, team ids uint16, new CarTelemetry2 packet id 16 carrying
active-aero X/Z mode and overtake mode). Packets consumed: Motion,
Session, LapData, Participants, CarSetups, CarTelemetry, CarStatus,
TimeTrial, CarTelemetry2.

Parsers are defensive where layouts are risky: Participants derives the
per-record size from the packet length (a layout change degrades to
skipping, not misparsing), and the Session-packet assist block rejects
values outside their documented ranges. Unknown `packetFormat` values
record a one-shot warning instead of guessing.

## Recording pipeline

The recorder keeps a **latest-value cache per car** for Motion,
CarTelemetry, CarStatus, CarTelemetry2, CarSetups, Participants and
TimeTrial. **LapData is the clock**: every LapData packet (60 Hz)
appends one sample row — the current cache values joined with that
packet's `currentLapTime`/`lapDistance` — to the active lap buffer of
each tracked car.

Tracked cars are only the player, the personal-best ghost and the rival
ghost; the PB/rival car indices come from the LapData packet trailer.
This is the ghost-harvesting trick: loading any leaderboard entry as a
Time Trial rival makes the game broadcast that lap's full telemetry on a
normal car slot.

A lap finalizes when the car's lap number increments (player) or when
its clock/distance rewinds (ghosts, which loop forever on lap 1 — see
[design-notes.md](design-notes.md) for every ghost rule and why it
exists). Finalization validates the buffer (enough samples, spans the
whole track, has a real lap time), dedupes repeating ghost laps by
`(role, lap_time_ms)`, then writes one `laps` row.

Player flashbacks rewind mid-lap: samples past the rewound position are
dropped and recording continues, so a flashback lap stays one lap.

Each sample stores: `t, d, x, y, z, spd, thr, brk, str, gear, drs, ot,
rpm, tfl, tfr, trl, trr, fuel, ers, aero` (time, lap distance, world
position, speed, inputs, DRS/overtake, tyre surface temps, fuel, ERS
charge, aero mode).

The recorder also maintains a status dict (`/api/status`) used by the
header chips and for self-diagnosis: packets/s, per-packet observed
sizes (layout probe), tracked-car buffer stats, ghost broadcast flags,
and the reason the last ghost lap was dropped.

## Storage

Two tables (`db.py`): `sessions` (one per game session UID: track,
session type, weather, packet format) and `laps` (role, lap number, lap
+ sector times, validity, tyre, top speed, car setup JSON, assist JSON,
team id, and the samples).

Samples are stored **column-oriented** — `{"t": [...], "d": [...], ...}`
— as zlib-compressed JSON in a single BLOB per lap (~100–200 KB → tens
of KB). Column orientation compresses far better than row tuples and is
exactly the shape the viewer wants; one decompress serves the whole lap.
Schema changes append columns via `_migrate()` (ALTER TABLE, additive
only); old blobs simply lack newer columns and the viewer treats those
as "not recorded".

## HTTP API

| Route | Returns |
| --- | --- |
| `GET /api/status` | recorder status dict (live) |
| `GET /api/tracks` | tracks that have laps, with lap counts + bests |
| `GET /api/tracks/{id}/laps` | every lap ever recorded on a track, all sessions |
| `GET /api/laps/{id}` | one lap: metadata + decompressed samples |
| `GET /api/laps/{id}/export` | the lap as a downloadable `.trace` file |
| `POST /api/import` | store an uploaded `.trace` file as a `guest` lap |
| `DELETE /api/laps/{id}`, `/api/laps/invalid`, `/api/sessions/{id}` | cleanup |

Responses are gzipped when the client accepts it. The viewer polls
`/api/status` once a second for the LIVE/IDLE chip, ghost indicators and
auto-refresh when a new lap lands.

A `.trace` file is one lap plus its session row as gzipped JSON
(`{"trace": 1, "session": …, "lap": …, "samples": …}`), meant to be sent
to another TRACE user and dropped onto their window. Import sanitises
the file — track/session/team names are re-derived from the internal id
tables, metadata is coerced to numbers, samples are validated — dedups
against laps already in the database (session uid + lap time + sample
count) and stores the lap under `car_role` `guest`, which the lap list
shows as a GUEST badge and files under the GHOSTS filter.

## Viewer

`app.js` is organised around a single `state` object (selected track,
lap A = viewed, lap B = reference, playhead time, view modes) and a
handful of subsystems:

- **Lap list** — all laps of a track across sessions, sortable
  (recent/fastest), filterable (you/ghosts, assists on/off, hide
  invalid), with per-session folding. Any lap can be viewed (A) and any
  other marked REF (B).
- **Track geometry** (`trackGeom`/`calibrateGeom`/`synthCoords`) — the
  map is always drawn from the bundled real circuit outline, never from
  telemetry, so shape/view/turn numbers are identical for every lap of a
  track. Per track, a one-time registration aligns the outline with the
  game: driving direction + start-line offset by curvature
  cross-correlation, then an affine game→outline fit (cached in
  localStorage). Lap samples are placed on the outline by lap distance,
  with the high-pass component of their true lateral offset re-applied so
  a zoomed map still shows the real driven line.
- **Corners** — detected from outline curvature; when the official turn
  count is known (`TURN_COUNT`), progressively weaker curvature
  thresholds fill the numbering so it matches the published numbers.
  Per-corner time deltas segment the lap at braking points (see
  design-notes).
- **Map rendering** — a static canvas (road ribbon, racing line colored
  by speed or by gap/dominance, sector gates, corner numbers, time-loss
  badges) redrawn only on zoom/resize/lap change, and a dynamic canvas
  (car + ghost dots) redrawn per frame. Scroll — on the map or on any
  chart — zooms into a corner; the map view is the single source of
  truth and the charts re-scale to its visible distance range.
- **Charts** — speed/throttle/brake/steer vs distance in per-channel
  colors, the reference lap overlaid as a grey ghost line, plus a
  delta-time trace (green/red fill = gaining/losing). Click/drag
  anywhere to seek; the playhead is shared with the map and HUD.
- **Halo HUD** — SVG instrument cluster (speed, gear, rev lights,
  throttle/brake arcs, steering, DRS/overtake/aero), draggable. The red
  card tags fold their cards in place (TELEMETRY, TIMING) or close them
  (SETUP); the steering section folds separately via its STEER pill.
- **Playback** — requestAnimationFrame loop advancing lap A's clock;
  the ghost dot shows where the reference lap was at the same elapsed
  time.

## Testing without the game

`tools/fake_game.py --speedup 40` replays the two bundled Melbourne laps
(`f1trace/demo.db`, the demo-mode laps) as authentic 2026-format UDP, with the
rival ghost reproducing the real game's shadow-car behaviour: shared
lap clock (parks at the line, rewinds on player restart), junk LapData
sectors, and CarTelemetry placeholder interleaving. If the recorder
mishandles a ghost quirk, the fake game catches it before a live session
does.
